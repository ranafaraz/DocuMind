# DocuMind - Offline Evaluation Results

Generated by `python -m evals.harness` over 180 deterministic documents (60 seeds x 3 doc types), no API keys, no downloads. Every document is scored against its **ground-truth record**.

## Headline: four configurations (pooled across doc types)

| Config | Field acc | Cell F1 | Doc exact | Validity | Total acc |
|---|---:|---:|---:|---:|---:|
| text | 0.573 | 0.962 | 0.300 | 0.911 | 0.911 |
| text + verify | 0.588 | 0.962 | 0.333 | 1.000 | 1.000 |
| layout | 0.984 | 1.000 | 0.911 | 0.911 | 0.911 |
| **layout + verify** | **1.000** | 1.000 | 1.000 | **1.000** | 1.000 |

Two effects, cleanly separated:

- **Layout geometry buys field-association accuracy.** Holding the verifier on, switching the reading-order extractor for the layout-aware one lifts field accuracy from 59% to 100% -- almost all of the gap is the two-column `form`, where reading order interleaves the columns (see per-doc-type table).
- **The schema verifier buys arithmetic validity.** Holding the extractor on `layout`, turning the verifier on lifts record validity from 91% to 100% and `total` accuracy from 91% to 100%, by reconciling the OCR-corrupted total against subtotal + tax.

## Null test: scramble the page geometry, re-run the layout extractor

| Config | Field acc | Cell F1 | Doc exact |
|---|---:|---:|---:|
| layout + verify (real geometry) | 1.000 | 1.000 | 1.000 |
| layout + verify (scrambled geometry) | 0.029 | 0.010 | 0.000 |

Destroy the layout and the geometry-based extractor collapses from 100% to 3% field accuracy -- confirming it was reading the page's geometry, not memorising text order.
